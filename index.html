  • Cleaning The System

    Overview

    Cleaning The System was a game I started working on right after I graduated. I graduated in 2020, which was the year of Covid, so looking for a job was something I decided to hold off on. I decided to build my portfolio instead. This game started as a game jam submission to the GMTK2020 game jam, but after it won top 20 out of 5,000 entries, I decided to develop it further.

    Gameplay

    The video above shows a few of the different mechanics in the game. The game's controls are only A and D, these can be remapped and there is also a mouse cursor input option. I wanted to stick with simplicity in the controls for this game. With that in mind, I came up with new mechanics that would work well with just two inputs as seen in the video above.

    Level Editor

    After making a few levels, I felt it was a little difficult to make levels inside Unity, so I made a level editor for myself. After working on it for a while, I figured this would be a great tool to include in the game, so I tried to make the editor as user friendly as I could. I did take a few courses on human-computer interaction at Texas A&M University, so I knew the basics of what a user would think. I also took a lot of ideas and feedback I received on my UI from previous games. The video above shows a little gameplay of the level editor. Alternatively, I created a time-lapse video of me creating a full level with my editor, this can be viewed here.

    This level editor is currently only in the Steam version of this game because I added Steam Workshop integration to allow players to easily create and upload their own levels directly to Steam. This allows other players to browser user created levels not only from in-game but also the Steam workshop. Downloading a user created level from the Steam application automatically loads the level into my game's files which is convenient for the player.

    Procedural Generation

    After I completed the Steam version of the game with the level editor. I decided to make a mobile version. I looked into the monetization options and found that most mobile games use the free with ads model. I didn't have a way to integrate ads into my game so I created a currency system for cosmetics as well as a procedurally generated level where you could earn those coins. With this I added an optional ad if players wanted to double the coins they obtained on a single run.

    The procedural map is a room based generation that looks at the current exit, gets a random room from my pre-built room list, and rotates and translates the room to fit nicely with the current flow. The video above shows a test run without any constraints. For my game, I needed a few constraints such as "never go directly up" and only rotate at a 45° angle from the exit.

  • Aw, Sheep!

    Overview

    This game was created as part of the Chillennium game jam, which lasts 48 hours, hosted at Texas A&M University. The premise of the game is to capture more sheep than your opponent before the sun goes down. In the end, the game turned out pretty well as we won the Best in Game Design and Fan Favorite awards.

    My Work

      • Multiplayer Split-Screen

      • UI Code and Design

      • Placing objects on spherical planet

      • Powerups and Spawners

      • Sunset + Sunset Timer

    More Information

    This game was created as part of the Chillennium game jam, which lasts 48 hours, hosted at Texas A&M University. The theme was "One for all and all for one". I was on a team with 3 others. We had 2 programmers and 2 artists. The premise of the game is to capture more sheep than your opponent before the sun goes down.

    I worked on the UI code and design, the spawners, and a scrapped path-finding system. The UI code and design involved the flow from the menu to the sun countdown. We did not have powerups initially, but I made the spawners in a way that takes prefabs, so creating one for powerups was easy to complete along with the sheep.

    I spent about 10 hours trying to get Unity's NavMesh to work on a sphere. I got it to work at one point by dynamically creating the NavMesh and rotating the NavMesh sphere( a separate sphere from the world, just for one sheep). As you can guess, having a new instance of the world for each sheep is not a good idea, so this was scrapped, and I moved on to do other parts. After the event, I got advice from some experts on how I could do spherical NavMesh. One idea I liked is to use the vertices of the sphere as navigation nodes.

    After the jam, our team spent some time coming up with ideas and fixes we would want for the game when we had the time to implement them. A lot of these were taken and implemented into the current version.

  • Fling For Freedom

    Overview

    This game was for a class at university. It was a semester-long project with me and three others. The goal of this game is to get the drone to the end of the level by throwing items, bumping off of walls, and moving back and forth in time to fix any mistakes.

    My Work

      • Timeline Mechanic

      • Time Control

      • Agile/Scrum Methodologies

    More Information

    This game was for Game Development(CSCE443) at Texas A&M University. I worked with three other programmers. Since we were all programmers, we decided to do simple 2D pixel art. One of the programmers was interested in it, so he became our artist.

    The instructor of this class pushed Agile/Scrum methodologies. Every two weeks, we would set what each of us would do, and at the end of the two weeks, we would show the class what we accomplished. I liked this because it made sure that each individual was doing their part.

    The goal is to get to the end of the level. There are traps and enemies preventing you from doing so. The catch is that you can control time. Using this time manipulation mechanic, you can avoid death or even come back from the dead.

    I mainly worked on the timeline mechanic. The system for it is not very complex. I store the position, rotation, velocity, and other important factors for every object that needs to be time-lined every frame. We capped the amount of time you have in a level, so we could use arrays which gave us a big boost in performance because we knew we would never have to create more data than what we initially set the time to.

    One of the tough parts of this project was creating a system that could handle all of this. I could timeline each object with just physics data, but we also had objects with animations and objects that affected other objects. To remedy this, I used inheritance. This way, I only have to traverse one list of objects that require timelining. In the unique cases like animations, I store the time manipulation functionality inside their overridden functions.

  • Wild West

    Overview

    This game was made by me and four others for a club-wide game jam. I was the sole programmer. This was the first time I had ever worked on a game for this long, a full semester. I learned about pathfinding with A* and turn-based and node-based gameplay.

    My Work

      • Turn-Based Gameplay

      • Grid-Based Gameplay

      • A* Pathfinding

    More Information

    This game was part of the TAGD(Texas A&M/Aggies Game Development Club) semester-long game jam. Our team consisted of me and 4 other members. Though, in reality, the game was made mainly by me and 1 of those other members, as the semester was a little too busy for the other 3 members to help out much. That 1 other member was a designer, and I was the sole programmer.

    Working on a game for a semester taught me a lot of things. The main ones being pace and scope. Since this was a semester-long game jam, we needed to slowly work on this throughout the semester. Trying to make the game all at once would only lead to burn out before the finish line. Our idea was to create a game like Fire Emblem, but we ended up with something not even close to it. We would all have times where we would be busy and would rather take our free time to relax, which is reasonable. So, I learned that even if we may have 3 months, the actual time to work on something might just be 3 weeks. With this, scope and pace should be taken into account. This led me to find a new appreciation for the Agile/Scrum methodologies.

    The node/tile-based and turn-based gameplay programming is what I mainly worked on for this project. I followed a tutorial series by Sebastian Lague for this project's AI pathfinding. This tutorial gave me a better understanding of A* than I had previously. This video series is also the one I always go to when I need a refresher on A*.

    Even though this project is dead, I did make changes after the jam. The gifs at the top show a few of those changes. I completely redid the UX with the feedback I received, making it easier for players to complete the actions they wanted to. A lot of the actions were behind unnecessary clicks that were removed to make it easier and faster for the player. We did not have an artist, so the UI might not look as good as it could. A lot of the art and animations were from third-party assets. In addition to these, the outline effect shown in the gifs was also from a third-party asset. The level shown in the gif was created by the designer that I worked with.

  • Shoot For The Stars

    Overview

    This game was created by me and 3 others for the 2 Color Game Jam. I mainly worked on the UI code and design. In this platformer, you control a gun with 1 function, to shoot.

    My Work

      • UI Code and Design

    3 friends and I decided to participate in a weekend game jam called the 2 Color Game Jam. The theme for the game jam was "Space". After hours of playing games and browsing Reddit, we stumbled upon a GIF where someone had replaced the main character in "Getting over it" with a gun.   • Bounce

    Overview

    The first game I ever completed. I did everything except the audio and the font. I published it to the Google Play Store(Currently removed). I learned a lot about Unity, programming, and creating a game.

    My Work

      • Gameplay

      • UI Code and Design

    More Information

    This was the first game I ever worked on. I didn't really know how hard game development was, so my initial concept for this game was to bounce lasers off reflective surfaces in 3D. I soon learned mirrors were expensive to render and that I could not model. With these constraints, I decided to make the concept about shooting a sort of energy ball and converted from 3d to 2D.

    This energy ball bounces off of the surfaces in the game. This is the core mechanic of the game. Using this, the player must get the green ball of energy from the green laser to the red laser.

    Apart from the audio, and the font, I created everything using Unity Docs/Forums, YouTube, and Stack Overflow as heavy references. The game doesn't have any complex code, but I still think it came out pretty well.

    Once it was done, I was able to publish it to the Android App Store. That said, the game is no longer available there, as I didn't keep up with it. I also did not market the game at all, so I didn't receive any feedback, but I did learn a lot about creating a game.

  • Priority List

    Overview

    I decided to try making my own calendar app using C# since I couldn't find a good one at the time.

    My Work

      • Notetaking Functionality

      • UI Code and Design

    More Information

    This is a combination of a calendar and a to-do list application.

    As a college student, I found that, I needed a way to keep track of all my homework and exams. I had been using the default Sticky Notes, but it was very difficult to make changes and organize my tasks. So I started searching for similar applications, but all I found were Sticky Note knock-offs, life-management applications, and Google Calendar. I liked Google Calendar's way of showing information, but I didn't like the process I had to go through to create a simple task. With all this in mind, I decided to create my solution.

    I didn't know how to make a standalone application, so I started searching how to make a simple sticky notes application since there were so many clones out there. I was led to WPF(Windows Presentation Foundation). The tutorial used Visual Basic, so I tried to understand it, but all I knew going into this project was C#, Python and some C++, so it was quite difficult. I then decided to do more research and found that I could use C#, which looked a lot more familiar since I used to to make my first game.

    I did some basic tutorials to learn how to create WPF applications with C#. Then I started creating my application. With the help of YouTube and Stack Overflow, I was able to make some great progress. At one point. I even thought about trying to sell it, but soon I found Google Keep and Microsoft To-Do and stopped working on it. That said, it helped me keep track of my work for the amount of time that I did use it.
